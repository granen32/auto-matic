# 피해야 하는 제안

현재 프로젝트에서 기본적으로 피해야 하는 패턴이다.

---

## 상태 관리

- 모든 상태를 Pinia 에 넣는 구조
- 서버 응답을 Pinia 에 그대로 캐싱하는 구조
- page 내부 임시 상태를 전역 상태로 끌어올리는 구조

---

## 컴포넌트 구조

- 하나의 `.vue` 파일에서 API, 검색, 리스트 선택, 다운로드, 상세 패널을 전부 처리하는 구조
- 템플릿 안에서 복잡한 계산과 조건 분기를 수행하는 구조
- 서버 응답 원본을 그대로 template 에서 직접 소비하는 구조

---

## 폴더 / 파일 구조

- PascalCase 또는 camelCase 파일명
- feature-first 대신 전역 `components/`, `apis/`, `stores/` 중심으로만 분리하는 구조
- 단순 유틸 composable 까지 전부 `.controller.ts` 로 만드는 방식
- 한 feature 가 다른 feature 내부 파일을 직접 import 하는 구조

---

## 라우팅

- 모든 라우트를 한 파일에 몰아넣는 것
- 정적 import 로 페이지 컴포넌트를 전부 등록하는 것
- 라우트 정의를 `pages/` 또는 `features/` 안에 흩뿌리는 것

---

## 그 외

- 현재 코드에 없는 기술을 이미 도입된 것처럼 가정하는 제안
- 작은 요구사항에 과도한 추상화와 제네릭을 도입하는 제안
- 무관한 영역까지 한 번에 같이 바꾸는 대규모 리팩토링 제안
