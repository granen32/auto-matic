# 피해야 하는 제안

Claude가 이 프로젝트에서 **기본적으로 피해야 하는** 패턴이다.
요청이 명시적으로 이 방향을 원하는 경우가 아니라면 다른 대안을 먼저 제시한다.

---

## 상태 관리

- 모든 상태를 Pinia에 넣는 구조
  - 페이지 한정 상태(검색어, 모달 열림 여부, 선택 row 등)는 controller 내부에서 처리한다.
- 서버 응답 데이터를 Pinia에 그대로 캐싱하는 구조
  - 서버 상태는 React Query에 맡긴다.

---

## 컴포넌트 구조

- 하나의 `.vue` 파일에서 API 호출, 필터, 다운로드, 상세 패널, 그리드 제어를 전부 처리하는 구조
- 템플릿 안에서 복잡한 조건문, 계산식, 데이터 가공을 수행하는 구조
- 서버 응답 원본을 그대로 template에서 직접 사용하는 구조 (필드명이 그대로 노출됨)

---

## 폴더 / 파일 구조

- 파일명을 PascalCase / camelCase로 만드는 구조
- feature보다 `components/`, `apis/`, `stores/` 전역 분리를 우선하는 구조
- 단순 유틸 composable까지 전부 `.controller.ts`로 만드는 방식
- 한 feature가 다른 feature 내부 파일을 직접 import하는 구조 (공유가 필요하면 `shared/`로 올린다)

---

## D3

- D3를 컴포넌트 내부에 직접 길게 그리는 구조
- Vue의 reactive 값을 D3 selection 안에서 계속 watch하면서 매번 전체 redraw하는 구조
- renderer 없이 SFC 안에서 axis, scale, transition을 모두 처리하는 구조

---

## 그 외

- 추측성 라이브러리 도입 제안 (현재 스택과 충돌하는 도구)
- 사소한 작업에 과도한 추상화 / 제네릭 / 헬퍼를 도입하는 제안
- 백워드 호환성을 위한 불필요한 alias / re-export 제안
- 한 PR에서 기능 추가와 무관한 영역까지 같이 리팩토링하는 제안
