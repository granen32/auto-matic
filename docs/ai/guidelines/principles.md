# 가장 중요한 원칙

이 프로젝트에서 Claude가 코드 제안 / 리팩토링 / 리뷰를 수행할 때 가장 우선해야 하는 원칙이다.

1. **화면 표현과 비즈니스 로직을 분리한다.**
2. **파일 및 폴더 이름은 모두 kebab-case를 사용한다.**
3. **비즈니스 흐름을 담당하는 composable은 `.controller.ts`를 사용한다.**
4. **Feature-first 구조를 우선한다.**
5. **Pinia는 공유 상태에만 사용한다.**
6. **React Query는 서버 상태 조회/캐싱에 우선 사용한다.**
7. **D3 로직은 Vue SFC 내부에 길게 작성하지 않고 renderer로 분리한다.**
8. **템플릿에는 복잡한 조건문과 비즈니스 계산을 넣지 않는다.**
9. **API 응답 원본을 UI 전역에 직접 퍼뜨리지 말고 API 계층 근처에서 정제한다.**
10. **페이지는 진입점, feature는 실제 기능 구현 위치로 본다.**

---

## 보충 설명

- "표현과 로직 분리"는 단순히 파일을 나누라는 의미가 아니라, **컴포넌트는 props/emits/렌더링에 집중하고 흐름은 controller에 둔다**는 의미다.
- `.controller.ts`는 단순 유틸 composable과 구분하기 위한 마커다. 비즈니스 흐름이 없는 hook성 composable에는 붙이지 않는다.
- feature-first가 우선이지만 정말로 여러 feature가 공유하는 것은 `shared/`로 올린다. 무리해서 feature 안에 가두지 않는다.
- "API 응답 정제를 API 계층 근처에서"는 백엔드 응답 필드명을 화면 곳곳에 그대로 흘리지 말라는 의미다. mapper/adapter를 api 폴더 주변에 둔다.
